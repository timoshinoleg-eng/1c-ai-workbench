# Бриф: фоновый демон индексации code-index (разделение writer/reader)

# История и обоснование пересмотра решения

В `claude-note-db` ранее было принято решение: «Не перестраивать архитектуру code-index (watcher-демон) — overkill для <10 проектов», с оговоркой «пересматривать при >50 проектах». Настоящий бриф **возвращается** к идее демона по другим причинам, не связанным с количеством проектов:

1. **Конфликт `serve.pid`** при попытке открыть вторую Claude Code-сессию на уже работающий проект — вторая сессия падает с «code-index: Failed». На практике это происходит постоянно (забытая вкладка VS Code, параллельные эксперименты, запущенный субагент).
2. **Watcher и индекс живут только пока жива Claude Code-сессия**. Каждый рестарт Claude Code заставляет заново делать initial reindex, тратить ЦП/RAM на уже сделанную работу и ждать до появления tools.
3. **N параллельных MCP-серверов тратят N-кратно ресурсы на один и тот же watcher**.

Количество проектов (<10 или 50) для этих болей не является определяющим фактором — они воспроизводятся и на одном активно используемом проекте. Переход к выделенному демону решает все три проблемы разом.

Этот абзац — **триггер пересмотра**: зафиксирован, чтобы при возможном откате решения в будущем можно было понять, какие именно боли его вызвали.

# Описание задачи

Переработать архитектуру проекта `code-index` (Rust, MCP-сервер для индексации кода), разделив её на два процесса:

- **Демон** — единственный на машине фоновый процесс, отслеживает список папок из своего конфига, поддерживает в них актуальные SQLite-индексы (`<path>/.code-index/index.db`). Единственный писатель.
- **MCP-сервер** — тонкий read-only слой, запускается per-сессия Claude Code (и другими клиентами). Только читает БД, ничего не индексирует, ничего не пишет. Сколько угодно экземпляров на один проект одновременно.

Сейчас эти роли совмещены: `code-index serve` в одном процессе совмещает MCP + watcher + writer. Это приводит к двум постоянным болям:

1. **Конфликт `serve.pid`** при запуске второй Claude Code-сессии на уже открытый проект — вторая сессия падает с «code-index: Failed».
2. **Watcher и индекс умирают вместе с сессией** — перезапустил Claude → ждёт initial reindex, тратит ЦП/RAM повторно на уже сделанную работу.

## Цель доработки

- Разрешить **произвольное число** параллельных MCP-читателей на один и тот же проект (Claude Code-сессии, VS Code-плагины, субагенты, CLI-запросы). Взаимные блокировки исчезают полностью — pidlock как per-project механизм становится не нужен.
- Индексация не зависит от жизненного цикла Claude Code-сессии: запустил Claude — уже свежий индекс, не надо ждать и не надо тратить ресурсы повторно.
- Ресурсы ЦП/RAM на watcher тратятся один раз на машину, а не N раз по числу открытых сессий.
- Сохранить кроссплатформенность (Windows/Linux/macOS) — не делать Windows-службу, использовать универсальный фоновый процесс + средства автозапуска каждой ОС.

## Требования к функционалу

### Демон

- **Кроссплатформенный Rust-binary** без Windows-specific API. Не Windows-служба.
- **Конфиг со списком папок**, редактируемый пользователем. Пути хранения:
  - Windows: `%APPDATA%\code-index\daemon.toml`
  - Linux/macOS: `~/.config/code-index/daemon.toml`
  Формат — TOML (стандарт в Rust-экосистеме).
- **Для каждой папки** — индивидуальная SQLite `<path>/.code-index/index.db`. Схема БД не меняется, режимы InMemory/Disk/Auto сохраняются.
- **File watcher через crate `notify`** (уже есть в проекте, `src/watcher.rs`). Debounce и batch настраиваемые — глобально или per-папка, значения по умолчанию — как сейчас (debounce 1500 мс, batch 2000 мс).
- **Hot-reload конфига по явной команде `code-index daemon reload`.** Демон перечитывает `daemon.toml`, сравнивает список папок с текущим: новые папки добавляются (status → `initial_indexing` → `ready`), удалённые прекращают отслеживаться (status → `not_started`). Автоматическое наблюдение за самим `daemon.toml` через notify — **не в MVP**, можно добавить позднее, если явная команда окажется неудобной.
- **Graceful shutdown** на SIGINT / SIGTERM / Ctrl-C: дописать текущий батч, закрыть БД, удалить PID-файл.
- **Логирование в файл**:
  - Windows: `%LOCALAPPDATA%\code-index\daemon.log`
  - Linux/macOS: `~/.local/state/code-index/daemon.log`
  Уровень логирования настраивается через конфиг и/или переменную `RUST_LOG`.
- **Один глобальный PID-файл демона** (не per-project): `%LOCALAPPDATA%\code-index\daemon.pid` / `~/.local/state/code-index/daemon.pid`. Защита от запуска второго экземпляра демона.
- **Обязательный health-endpoint** (глобальное правило: у любого MCP/сервисного процесса с API должен быть health). Канал: loopback HTTP, named pipe на Windows или Unix socket на POSIX. Метод `health` возвращает:
  - Общий статус демона: `running`, uptime, версия, PID, дата старта.
  - **Per-path статус** каждой отслеживаемой папки со значениями:
    - `not_started` — папки нет в конфиге, либо демон не начал её обрабатывать
    - `initial_indexing` — идёт первичная индексация
    - `ready` — индекс актуален, watcher активен
    - `reindexing_batch` — watcher обрабатывает батч изменений
    - `error` — ошибка, подробности в поле `error_message`
  - Для статусов `initial_indexing` и `reindexing_batch` — прогресс `{files_done, files_total, percent}`.
- **Первый запуск папки** (добавлена в конфиг впервые, `.code-index/index.db` отсутствует): демон создаёт БД, выставляет статус `initial_indexing`, запускает full reindex. По завершении — переход в `ready`, запуск notify-watcher.
- **Последующий запуск демона** (БД уже есть): full reindex с mtime fast-path (v0.4.0, уже реализован в `src/indexer/mod.rs`) — обрабатывает только изменившиеся файлы; короткая фаза `initial_indexing` и быстрый переход в `ready`.
- **Переход статуса во время работы**: watcher ловит изменение → папка переходит в `reindexing_batch` → после записи батча в БД → обратно в `ready`.

### MCP-сервер (переработка)

- **Полная замена текущего режима `serve`**: старый `serve` удаляется (возможно, с релизом-прокладкой, который печатает deprecation-warning). Новый MCP — тонкий read-only слой.
- БД открывается в режиме `SQLITE_OPEN_READ_ONLY`. Никакого writer, никакого watcher, никакой инициализации схемы.
- **Сколько угодно параллельных экземпляров MCP на один и тот же проект** — штатный сценарий. Никаких взаимных блокировок, никаких pidlock-конфликтов.
- **Проверка статуса папки перед tool-call** (IPC к демону — миллисекунды, кешируется между tool-call'ами с коротким TTL):
  - `ready` → обычное чтение БД и ответ.
  - `initial_indexing` / `reindexing_batch` → структурированный ответ `{status: "indexing", progress: {files_done, files_total, percent}}`. Tool **не падает**, клиент (Claude Code, агент) может показать прогресс и повторить позже.
  - `not_started` → ответ с подсказкой «добавьте путь в daemon.toml и перезагрузите демон».
  - Демон недоступен (health не отвечает) → ответ `{status: "daemon_offline", message: "Фоновый демон code-index не запущен. Запустите 'code-index daemon start' или убедитесь, что Scheduled Task активна."}`. MCP **не падает** и остаётся подключённым — пользователь сможет продолжить работу после запуска демона без рестарта Claude Code.
- **Текущий per-project pidlock** (`.code-index/serve.pid`) удаляется полностью — больше не нужен ни для чего, потому что MCP не пишет.
- **Собственный health-инструмент MCP** (уже есть — сохранить), плюс поле «демон жив?» в его ответе: MCP при своём health-запросе проксирует запрос к health-endpoint демона и возвращает статус обоих.

### CLI-команды чтения

- `query`, `stats`, `search-function`, `get-function`, `grep-body`, `get-callers`, `get-callees`, `get-imports`, `get-file-summary`, `search-text` и т.д. — уже работают через `SQLITE_OPEN_READ_ONLY`, изменения не требуются.

### CLI-команды управления демоном

Новые подкоманды (команды в дереве `code-index daemon` или отдельный binary `code-index-daemon` — выбрать в процессе проектирования):

- `code-index daemon start` — запустить демон (возвращает сразу, PID пишется в лог)
- `code-index daemon stop` — попросить демон завершиться (SIGTERM + таймаут)
- `code-index daemon status` — напечатать health-ответ в человекочитаемом виде (таблица с per-path статусами и прогрессом)
- `code-index daemon reload` — перечитать конфиг без рестарта (если выбран этот механизм hot-reload)
- `code-index daemon logs` (опционально) — tail лог-файла демона

### Автозапуск демона

- **Windows:** Scheduled Task при входе пользователя. Скрипт установки (`install-daemon-autostart.ps1` или аналог) создаёт задачу с абсолютным путём к `code-index.exe`, запускает с правами пользователя, не требует повышения.
- **Linux:** systemd user unit (`~/.config/systemd/user/code-index.service`). Запуск при логине пользователя. Инструкция по установке в документации.
- **macOS:** launchd user agent (`~/Library/LaunchAgents/com.user.code-index.plist`). Запуск при логине.

В рамках MVP реализуется и документируется Windows-вариант (основная целевая платформа пользователя). Linux и macOS — описываются в документации как будущая работа.

## Алгоритм работы (желаемое состояние)

1. Пользователь устанавливает `code-index`, редактирует `daemon.toml` со списком папок. Запускает скрипт установки Scheduled Task (Windows) / systemd unit (Linux) / launchd plist (macOS).
2. Scheduled Task запускает `code-index daemon start` при логине пользователя.
3. Демон:
   - Читает конфиг, по каждой папке: либо открывает существующий `.code-index/index.db`, либо создаёт новый.
   - Выставляет статус папки в `initial_indexing`, делает full reindex с mtime fast-path (пустая БД → все файлы; существующая БД → diff по mtime).
   - По завершении переводит папку в `ready`, запускает notify watcher.
   - Пишет PID-файл, открывает health-IPC.
4. Claude Code-сессия стартует → MCP-сервер подключается к БД нужного проекта в read-only → первым делом спрашивает у демона статус папки:
   - `ready` → сразу работает.
   - `initial_indexing` → tools отвечают `{status: indexing, progress: {...}}`, клиент может пробовать снова.
   - Демон недоступен → `{status: daemon_offline}` с подсказкой.
5. Пользователь редактирует файл → notify ловит событие в демоне → папка переходит в `reindexing_batch` → debounce 1500 мс → парсинг и запись батча → обратно в `ready`. MCP-читатели при следующем tool-call сразу видят новое состояние индекса.

## Совместимость и миграция

- Режим `serve` **полностью удаляется** (решение пользователя).
- Существующие каталоги `.code-index/` и БД остаются работоспособны — схема не меняется.
- Предлагается промежуточный релиз: в версии N-1 команда `serve` печатает deprecation-warning и ссылку на документацию о миграции, в версии N — удаляется.
- `.mcp.json` во всех проектах пользователя (RepoUT, RepoBP_1, RepoBP_2, claude-note-db, code-index, MCP-Servers, и т.д.) — автоматическая миграция не нужна: MCP-сервер по-прежнему вызывается как внешний процесс, меняется только внутреннее поведение.

## Риски и ограничения

- **Кроссплатформенность notify.** Crate `notify` имеет разную надёжность на разных ОС:
  - Linux inotify — лимит на число watches (настраивается в `/proc/sys/fs/inotify/max_user_watches`).
  - macOS FSEvents — coalescing событий может скрыть часть изменений, fallback на polling может понадобиться для больших деревьев.
  - Windows ReadDirectoryChangesW — работает стабильно, но известны проблемы с сетевыми папками.
  - Нужно продумать fallback на polling для проблемных ФС (через конфиг).
- **Scheduled Task стартует при логине, а не при загрузке ОС.** Если пользователь не вошёл в систему, демон не работает. Приемлемо для dev-машины пользователя, не приемлемо для сервера.
- **PATH в Scheduled Task урезан** (известная проблема Windows, зафиксирована в `~/.claude/CLAUDE.md`): команды из интерактивной консоли могут быть недоступны. Скрипт установки должен прописывать **абсолютный путь** к `code-index.exe` в параметрах задачи.
- **Размер БД в режиме InMemory.** Текущий режим `Auto` загружает БД в RAM, если хватает места. Для демона, который держит индексы **всех** отслеживаемых папок одновременно, суммарный RAM-след может стать большим. Нужно пересмотреть дефолт `memory_max_percent` в контексте общего демона, возможно — per-папка `storage_mode` в `daemon.toml`.
- **Обратное решение.** Предыдущая карточка в `claude-note-db` фиксировала «overkill для <10 проектов». Настоящий бриф это решение пересматривает — см. раздел «История и обоснование пересмотра решения» в начале брифа. После внедрения завести новую карточку в `claude-note-db`, которая отменит старую и зафиксирует новые причины.
- **Атомарность записи БД** при переиндексации. Сейчас в serve-режиме MCP-сервер сам и пишет, и читает — вопрос атомарности решён транзакциями. При разделении: MCP читает в момент, когда демон пишет → SQLite WAL справляется, но нужно проверить, что read-only-подключение MCP видит консистентное состояние после commit'а демона.

## Верификация (как проверить end-to-end)

1. Собрать проект в release-режиме. Запустить демон: `code-index daemon start`. Убедиться, что `code-index daemon status` показывает все папки из `daemon.toml` со статусом `ready` после завершения initial reindex.
2. Запустить **две** Claude Code-сессии на один и тот же проект (например, `C:\RepoUT`) параллельно. Убедиться, что в обеих MCP-сервер `code-index` показан как **Connected**, без «Failed». Сейчас вторая сессия падает по pidlock.
3. Отредактировать файл в `C:\RepoUT` (добавить/изменить функцию). Через ~2 секунды вызвать `grep_body` или `get-function` по новому содержимому в **обеих** сессиях — возвращается обновлённый индекс.
4. Убить процесс демона (`taskkill /F /IM code-index.exe` или соответствующий MCP-tool демона). Убедиться, что MCP-tools в открытой Claude Code-сессии продолжают работать (возвращают по старому индексу), но новые изменения в файлах не подхватываются. Health-tool MCP возвращает `{daemon: offline}`.
5. Снова запустить демон. Убедиться, что изменения, сделанные за время простоя, подхватываются в следующие 1–2 секунды после старта (mtime fast-path срабатывает).
6. Перелогиниться в Windows. Проверить `tasklist /FI "IMAGENAME eq code-index.exe"` — демон автоматически запущен Scheduled Task'ом.
7. Удалить папку из `daemon.toml` и вызвать `code-index daemon reload` (либо рестарт, если выбран этот механизм). Убедиться, что: (а) демон больше не следит за этой папкой, (б) её `.code-index/index.db` не удаляется автоматически (это решение пользователя), (в) MCP для этой папки возвращает `{status: not_started}`.
8. Первый запуск новой папки: добавить новый путь в `daemon.toml`, перезагрузить демон. Сразу же попытаться дёрнуть MCP для этой папки. Ожидается: tool отвечает `{status: indexing, progress: ...}`, через какое-то время — статус становится `ready` и tools начинают возвращать результаты.
