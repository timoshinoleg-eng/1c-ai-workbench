## ADR: 004 — Workbench Auto-Detection

**Статус:** accepted

**Контекст:**
Cockpit (`tools/cockpit-app`) требует корневой каталог `1c-ai-workbench` для запуска
MCP-серверов. Изначально путь был захардкожен в `C:\1c-ai-workbench`, что ломало
чистую установку (bug report `2026-06-24-cockpit-config-defaults.md`, P0).
Нужна эвристика автообнаружения workbench на первом запуске без ручного
редактирования `%APPDATA%\1c-ai-cockpit\config.json`.

**Рассмотренные варианты:**
1. Вариант A — оставить hardcoded путь и полагаться на инструкции пользователя.
2. Вариант B — спрашивать путь в onboarding вне зависимости от окружения.
3. Вариант C — runtime auto-detection с валидацией по marker-файлу
   `tools\code-index-mcp\target\release\bsl-indexer.exe`.

**Решение:** Вариант C — эвристика в `tools/cockpit-app/src-tauri/src/config.rs`.

Порядок кандидатов (`config.rs:workbench_candidates`):
1. Переменная окружения `WORKBENCH_ROOT` (operator override).
2. Соседние каталоги относительно `cockpit-app.exe`:
   - `<exe_dir>`, `<exe_dir>\workbench`, `<exe_dir>\1c-ai-workbench`
   - `<parent(exe_dir)>\workbench`, `<parent(exe_dir)>\1c-ai-workbench`
3. Распространённые абсолютные пути:
   - `C:\1c-ai-workbench`
   - `C:\Program Files\1c-ai-workbench`
   - `%LOCALAPPDATA%\1c-ai-workbench`
   - `%USERPROFILE%\1c-ai-workbench`

Валидация (`config.rs:is_valid_workbench_root`):
- каталог существует;
- внутри него есть `tools\code-index-mcp\target\release\bsl-indexer.exe`.

Первый прошедший валидацию кандидат становится `workbench_path` по умолчанию.
Если ни один не подошёл, поля `workbench_path`, `code_index_home`, `help_db_path`
остаются пустыми, и onboarding должен предложить выбрать путь вручную.

**Обоснование:**
Эвристика покрывает основные сценарии развёртывания: dev-окружение
(`C:\1c-ai-workbench`), bundled install (workbench рядом с `.exe`) и
per-user install (`%LOCALAPPDATA%`). Marker-файл отбрасывает ложные совпадения
по имени каталога. Реализация остаётся в одном Rust-файле и не требует
изменений в UI до тех пор, пока автообнаружение не даёт сбоев.

**Последствия:**
- `CockpitConfig::default()` больше не содержит жёстко закодированных путей.
- Добавлена команда `validate_config` (`commands.rs`) с чеклистом проверки
  workbench layout.
- Операционный runbook обновлён: свежая установка теперь ожидает
  нажатия `Проверить пути` и, при необходимости, ручного выбора workbench.

**Известные gaps, приоритеты и митигации:**

| Приоритет | Gap | Митигация | Статус |
|---|---|---|---|
| **P0** | `WORKBENCH_ROOT` или любой найденный каталог с `bsl-indexer.exe` может быть подставлен злоумышленником. Сейчас проверяется только наличие файла, но не его подпись, хэш, архитектура или версия. | Доверять `WORKBENCH_ROOT` только если она установлена известным инсталлятором; добавить проверку цифровой подписи / хэша бинарника; если путь пришёл из env var или нестандартного расположения — показывать warning и требовать ручного подтверждения. | **Требуется follow-up bug** |
| **P0/P1** | Если на машине несколько валидных установок workbench, текущий код выбирает **первую попавшуюся** в фиксированном порядке кандидатов. Поведение детерминировано, но может быть неожиданным для пользователя (например, устаревшая dev-копия в `C:\1c-ai-workbench` побеждает свежую копию рядом с `.exe`). | Сохранить детерминированный порядок, но отдавать приоритет пути, прописанному инсталлятором (registry / companion metadata); если найдено более одного валидного кандидата — показывать chooser в onboarding; логировать выбор. | Рекомендуется к реализации |
| **P1** | `bsl-indexer.exe` может существовать, но иметь неподходящую архитектуру (x86 vs x64) или устаревшую версию. Сейчас проверяется только `is_file()`. | Добавить version check: запускать `bsl-indexer.exe --version` (или читать PE-метаданные) и сверять с `Cargo.toml` / минимально допустимой версией. При несовпадении помечать workbench как невалидный. | Рекомендуется к реализации |
| **P2** | WSL-контуры `\\wsl$\...\1c-ai-workbench` не входят в список кандидатов. Rust `PathBuf` корректно парсит UNC-синтаксис, и `Path::is_dir()` работает, если путь смонтирован, но автопоиск его не найдёт. | Задокументировать как known limitation. Пользователям WSL нужно явно задавать `WORKBENCH_ROOT` или выбирать путь вручную. | Known limitation |
| **P2** | Symlinks / junctions. Проверено локально: junction (`mklink /J`) к каталогу с `bsl-indexer.exe` проходит `is_valid_workbench_root() == true`; `Path::join(...).is_file()` разрешает junction. Символические ссылки (`mklink /D`) требуют прав администратора, но семантически Rust обрабатывает их аналогично. | Дополнительных мер не требуется; canonicalизация не выполняется, что приемлемо, пока marker-файл достижим. | Проверено, ограничение задокументировано |
| **P2** | Пробелы и кириллица в пути. Проверено локально: пути вида `C:\tmp\adr-test\Имярек\1c-ai-workbench` и `C:\tmp\adr-test\adr test\1c-ai-workbench` проходят валидацию. Rust использует WTF-8/`OsString` для Windows-путей; переменные окружения транскодируются runtime. | Дополнительных мер не требуется. | Проверено, ограничение задокументировано |

**Рекомендации по follow-up:**
1. **P0 — безопасность автообнаружения:** завести баг и реализовать проверку
   происхождения `WORKBENCH_ROOT` (installer provenance) + верификацию
   `bsl-indexer.exe` перед первым запуском. Без этого автообнаружение остаётся
   vector для подмены бинарников.
2. **P1 — version/arch check:** добавить до GA v1, так как несовпадение версий
   workbench и Cockpit приводит к неявным ошибкам `initialize` в MCP.
3. **P1/P2 — chooser и WSL:** отложить до запроса enterprise-пользователей;
   текущий heuristic покрывает Windows-контуры, описанные в операционном runbook.
